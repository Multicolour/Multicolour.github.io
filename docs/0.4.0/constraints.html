Constraints

Constraints are a simple, “logic free” way to limit behaviour on your database without writing any code or custom routes.

Constraints come in 3 formats:

  • key: String
  • key: { compile: Boolean, value: Any }
  • key: Function(request, constraints) -> Object

When constraints are found on your blueprint, it will by default check a few things to calculate best what you want to do.

A simple way to view what checks occur is:

if there are constraints
          does it have a compile option?
            if it's false, use the value as it appears, e.g false, "awesome"
            if it's true, access the property as it appears from the request object
          if the constraint is a function, run the function; bound to the collection, with the request and return the constraint.
          if the constraint doesn't have a compile option
            access the property as it appears from the request object

The most common constraint is ensuring a user has sufficient privileges to read|write something and would come in the form of

{
          user: "auth.credentials.user.id"
        }

The way this works is the String is a path to an object on the request object, this is in-fact setting the constraint to request.auth.credentials.user.id.

Verbs and constraints

You must specify your constraints on all the http verbs you wish to constrain. The verbs must appear as lowercase.

Example:

...
        constraints: {
          get: { public: { compile: false, value: true } },
          post: { user: "auth.credentials.user.id" },
          put: { user: "auth.credentials.user.id" },
          delete: { user: "auth.credentials.user.id" },
          patch: { user: "auth.credentials.user.id" }
        },
        ...
